// 03. 实战流程

5 步让 AI 帮你写代码

这是一套经过验证的实战流程，适用于任何 AI 编程工具。 掌握这 5 步，你就能高效地与 AI 协作。

01

明确需求

清晰描述你想要实现什么。越具体，AI 的输出越精准。

示例 
❌ 模糊的描述：
"帮我写个网络程序"

✓ 清晰的描述：
"用 C 语言实现一个高性能 TCP 服务器：
- 使用 epoll 实现多路复用
- 支持至少 10000 并发连接
- 每个连接需要独立的读缓冲区
- 目标平台是 Linux (CentOS 7)"
💡 技巧：说明语言、平台、性能要求、约束条件
02

选择工具

根据任务类型选择合适的 AI 工具。

🔧 完整功能开发

Claude Code / Cursor

📝 快速函数补全

GitHub Copilot

🤔 架构讨论

Claude.ai / ChatGPT

⚙️ 脚本/运维

AI CLI 工具

03

投喂上下文

让 AI 了解你的项目背景。上下文越充分，结果越准确。

📁 项目结构

告诉 AI 你的项目是如何组织的

📄 相关代码

提供需要参考或修改的现有代码

📋 编码规范

说明命名风格、错误处理方式等

⚠️ 约束条件

性能要求、兼容性、安全限制

cd ~/my-project

04

生成代码

AI 生成代码后，你需要审查和验证。

✓ 检查逻辑是否正确
✓ 验证边界条件处理
✓ 确认内存安全（C 语言）
✓ 检查错误处理完整性
✓ 验证是否符合编码规范
⚠️
重要提醒

AI 生成的代码不是 100% 正确的。作为资深开发者，你的审查是必不可少的环节。

05

调试优化

遇到问题？让 AI 帮你调试。迭代优化直到满意。

调试对话示例 
你: 代码编译报错了，错误信息是：
    "undefined reference to 'epoll_create1'"

AI: 这个错误说明你的系统 epoll 支持可能有问题。
    请检查：
    1. 内核版本 (需要 2.6.27+)
    2. 编译时是否正确链接

    尝试在编译命令中添加 -lrt 参数：
    gcc -o server server.c -lrt

你: 加了 -lrt 还是报错

AI: 让我看看你的代码... 问题出在头文件。
    epoll_create1 需要 _GNU_SOURCE 宏。

    在文件开头添加：
    #define _GNU_SOURCE
    #include <sys/epoll.h>
提问 → 获得解答 → 尝试 → 反馈结果 → 继续优化
// 04. 场景示例

C/云计算实战场景

这些是 C/云计算开发者最常遇到的场景， 每个场景都附有可直接使用的提示词模板。

场景描述

有一个处理网络数据包的函数，在高并发下性能不够理想，需要优化。

提示词模板

请帮我优化这个 C 函数的性能：

[code here]

当前问题：
- 在 10k 并发下 CPU 使用率过高 (80%+)
- 函数被频繁调用，是热点路径

优化目标：
- 降低 CPU 使用率
- 减少内存分配次数
- 保持代码可读性

环境信息：
- 目标平台：Linux x86_64
- 编译器：GCC 9+
- 可使用 SIMD 指令

AI 通常会建议

  • 减少内存分配（预分配、对象池）
  • 避免不必要的拷贝（指针传递）
  • 使用更高效的数据结构
  • SIMD 向量化
  • 缓存友好的内存布局

场景描述

需要编写一个自动化脚本，监控服务器状态并在异常时发送告警。

提示词模板

请帮我写一个服务器监控脚本：

需求：
1. 监控 CPU、内存、磁盘使用率
2. 监控指定进程是否存活
3. 监控 TCP 端口是否可连接
4. 异常时通过企业微信发送告警

技术要求：
- 使用 Bash 或 Python
- 兼容 CentOS 7 / Ubuntu 20.04
- 低资源占用
- 支持配置文件

输出要求：
- 完整脚本代码
- 配置文件模板
- systemd service 文件

场景描述

有一个复杂的 C 模块，需要为其他团队生成 API 文档。

提示词模板

请为以下 C 代码生成 API 文档：

文档要求：
1. 每个公开函数的说明
2. 参数说明（类型、含义、约束）
3. 返回值说明
4. 使用示例
5. 线程安全性说明

格式要求：

场景描述

有一段 10 年前的 C 代码，需要重构以提高可维护性和安全性。

提示词模板

请帮我重构这段 C 代码：

[code here]

重构目标：
1. 提高代码可读性
2. 修复潜在的内存安全问题
3. 添加适当的错误处理
4. 拆分过长的函数
